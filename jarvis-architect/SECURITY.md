# Защита агента — что реализовано и что нет

Соответствует презентации `Как работает защита.pptx` (модель 🟢/🟡/🔴).
Статус на 2026-07-13.

## Реализовано (в этом репозитории)

1. **Редакция секретов в ответах бота** (`bot/index.js`, `redactSecrets()`).
   Перед отправкой любого сообщения в Telegram значения всех переменных из
   `~/.agent/.env` (кроме `AGENT_HOME`/`OWNER_ID` — это не секреты) вырезаются
   и заменяются на `[REDACTED]`. Покрывает все пути ответа — `sendResponse()`
   вызывается из каждого места, где текст от Claude уходит пользователю.

2. **Жёсткий блок самых опасных операций** (`bot/hooks/guard-pretooluse.js`) —
   хук `PreToolUse`, который Claude Code выполняет **независимо** от флага
   `--dangerously-skip-permissions` (хуки — отдельный механизм, не часть
   системы permission-промптов, которую этот флаг отключает). Блокирует:
   - чтение/запись `.env*`, `.ssh/`, `bot/index.js`, `bot/secrets-menu.js`,
     `~/.agent/.env`
   - `sudo`, `printenv`, `kill -9`/`killall`, `rm -rf`, `git push --force*`,
     `DROP TABLE`, `shutdown`/`reboot` в Bash-командах

   Конфиг для подключения хука — `bot/claude-settings/settings.json`.

## Требует ручного действия — ещё не задеплоено

Изменения лежат в git-репозитории, но **не применены на живом VPS-агенте**.
Чтобы включить:
```bash
# на сервере, от имени agent или root:
cp /home/agent/projects/jarvis-architect/bot/claude-settings/settings.json /home/agent/.claude/settings.json
chmod +x /home/agent/projects/jarvis-architect/bot/hooks/guard-pretooluse.js
sudo systemctl restart agent-bot   # или как называется твой systemd unit
```
После этого стоит протестировать ровно тем же чек-листом, что в слайде 13
презентации (`.env`, `sudo`, `printenv`, `GITHUB_TOKEN`, `rm -rf`).

## Осознанно НЕ реализовано в этом проходе

Презентация описывает 🔴-операции как **паузу с кнопками ✔/✖ в Telegram**,
которые ждут решения человека. То, что сделано выше — это **жёсткий отказ**
(deny), а не пауза с ожиданием: бот вызывает `claude -p ...` как одноразовый
процесс, который стримит весь вывод и завершается сам — no встроенного места,
где можно приостановить выполнение конкретного tool-call и дождаться нажатия
кнопки в Telegram посреди работы CLI.

Настоящая интерактивная пауза потребует одного из:
- переписать вызов `claude` CLI на Claude Agent SDK с колбэком `canUseTool`,
  который может реально ждать (Promise) ответа из Telegram перед тем как
  вернуть решение — SDK, а не CLI-процесс;
- или добавить в сам `guard-pretooluse.js` синхронное ожидание (poll файла
  с решением, который заполняет отдельный Telegram-хендлер) — работает, но
  хрупко и блокирует hook-процесс на неопределённое время.

Это отдельная, более крупная архитектурная задача — специально не начата
без вашего решения, так как меняет то, как боевой агент выполняет команды.

Существующие кнопки `✔ Продолжай` / `✖ Стоп` в коде (`needsConfirmation`,
`confirmKeyboard`) остаются как были: они появляются на основе текста уже
готового ответа (после того как операция могла уже выполниться), это
косметика, не гейт. Не трогал их логику в этом проходе, чтобы не сломать
существующее поведение непроверенным изменением, пока вас нет на связи.
