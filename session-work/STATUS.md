## 2. Транскрибация видео `соцсети 1.avi` — В ПРОЦЕССЕ (фон)
Пользователь попросил (2026-07-15) вытащить из `C:\Users\HP\Desktop\соцсети 1.avi`

**Как устроено:**
- Аудио уже извлечено (mono 16kHz wav) — но лежит в session-scratchpad, который
  может не пережить рестарт чата. Если `socseti1.wav` пропал — просто извлеки
  заново командой (быстро, ~1 минута):
  ```
  ffmpeg -y -i "C:\Users\HP\Desktop\соцсети 1.avi" -vn -ac 1 -ar 16000 -acodec pcm_s16le "<новый путь>\socseti1.wav"
  ```
  (ffmpeg уже установлен через winget: `Gyan.FFmpeg`)
- Транскрибация идёт через `faster-whisper`, модель `"base"`, `device="cpu"`,
  `compute_type="int8"`, `language="ru"`, `vad_filter=True` — скорость примерно
  реального времени (не быстрее). Модель `"small"` пробовали первой — в 3+ раза
  медленнее, не годится, переключились на `"base"`.
- **ВАЖНО:** на этой машине системный прокси (Happ VPN) регистрируется как
  `socks=` в реестре Windows, что `httpx`/huggingface_hub автоопределяют как
  `socks4` (не поддерживается) → `ValueError`. Перед запуском ЛЮБОГО
  Python-скрипта, качающего что-то из интернета (включая veci whisper-модели),
  нужно явно выставить:
  ```
  export HTTPS_PROXY="socks5://127.0.0.1:10808"
  export HTTP_PROXY="socks5://127.0.0.1:10808"
  ```
  См. также этот же баг в `C:\Users\HP\Jurist\llm.py` (постоянная проблема на
  этой машине, не специфичная для транскрибации).
- Скрипт транскрибации: копия сохранена в
  `C:\Users\HP\Documents\Project\session-work\transcribe_script.py` (та же
  версия, что использовалась). Пишет прогресс построчно с таймкодами
  (`[123s] текст...`), можно смотреть на лету.
- **Живой лог прогресса синхронизируется сюда:**
  `C:\Users\HP\Documents\Project\session-work\socseti1_transcribe.log`
  (копия из session-scratchpad, обновляется при каждой проверке — смотри
  таймкод последней строки, чтобы понять текущий прогресс в секундах аудио;
  всего аудио ~8880 секунд).

**Если процесс транскрибации мёртв (проверить: `tasklist | grep python`,
искать процесс с ~400-500МБ+ памяти) и лог не растёт — перезапустить:**
```bash
cd "C:\Users\HP\Documents\Project\session-work"
export HTTPS_PROXY="socks5://127.0.0.1:10808"
export HTTP_PROXY="socks5://127.0.0.1:10808"

```
Транскрибация начнётся с нуля (whisper не поддерживает резюме с середины
файла) — это нормально, модель быстрая (~real-time), не критично.

**Когда транскрипт завершится** (в логе появится строка `DONE`): прочитать

**Новая гипотеза (2026-07-15 ~23:00):** обрыв происходит на этапе TLS-
хендшейка (ошибка `_ssl.c:1063: handshake operation timed out` — TCP уже
установлен, рвётся именно негоциация TLS), причём **только для
api.anthropic.com**, а не для google.com/ya.ru (которые проверяются в цепочке
раньше и проходят). Это указывает на точечные потери пакетов на пути
VPN-туннель → конкретный Cloudflare PoP, к которому anycast приводит
api.anthropic.com — а не на общую проблему VPN-роутинга.
Проверено: `curl -4` к api.anthropic.com сейчас проходит нормально (CF-RAY
показал дата-центр ORD/Chicago). `curl -6` не резолвит вовсе — но это
системная особенность (машина не имеет реального IPv6-маршрута), не
специфично для Anthropic, скорее всего не связано с обрывами (обрывы — это
handshake timeout, а не resolve failure).

**Следующий шаг, если продолжать:** поймать обрыв в реальном времени и
снять `tracert`/`mtr` до 160.79.104.10 в момент сбоя, либо посмотреть, меняет
ли переключение VPN-сервера/протокола в Happ частоту обрывов. Не доказано
на 100% — рабочая гипотеза, не подтверждённый факт.
